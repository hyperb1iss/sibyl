# Sibyl Architecture Audit — Post-Cleanup

**Date:** 2026-05-28
**Branch:** `main` (HEAD `ebc3c40b`)
**Scope:** Whole codebase — DB/driver layer, SurrealDB, graph, retrieval, services, persistence,
API/MCP, CLI, web. Focused on the four questions: is the architecture intentional, how much
boilerplate, how do we manage SQL/queries, and are there performance issues.
**Methodology:** Parallel swarm of 13 specialist reviewers (7 cross-surface lenses —
intentionality, boilerplate, query-management, performance, security/multi-tenancy,
migration-completeness, testing/observability — and 6 depth auditors on the riskiest surfaces).
Every critical/high finding was then independently adversarially verified against current code (15
confirmed, 3 demoted as overstated). The three headline claims (the per-org lock, the RBAC
fallback, the facade indirection) were additionally hand-checked against source. 31 agents, ~2.6M
tokens.

> This audit is a **delta against `AUDIT_2026-05-05.md`**, which was written mid-migration. Since
> then a large cleanup landed: Graphiti and the Postgres/relational tail are gone from deps and
> code, the monolithic `driver.py` was decomposed, schema versioning was added, and `native_*`
> identifiers were collapsed to plain names. Several of the prior audit's scariest findings are now
> **fixed** — those are called out explicitly so we don't keep relitigating them.

> **Revised after Codex peer review (gpt-5.5, xhigh).** Codex independently verified the findings
> against source. It confirmed the load-bearing ones (C1, H1/H2, H3, H5, H6, and the RBAC / MCP-JWT
> / API-key-scope "fixed" claims), corrected two overstatements, and surfaced two new high-severity
> misses — all integrated below. Corrected: the `surrealkv://` "fixed" claim (it's default-deny with
> an explicit opt-in, and core `CoreConfig` is still unguarded — not a categorical rejection); the
> `bm25`/`reranking` "dead code" claim (`bm25` is exported but production-unused; `reranking` is a
> *wired optional* path missing its dependency). Added: **H10** (API-key empty-scope MCP
> auto-promotion) and a checked-execute-results requirement folded into **C1**.

## Executive verdict

The migration **landed in the engine room but not the front office**. `sibyl-core` is now a
genuinely intentional, one-way-layered library with real abstractions and DB-native query
discipline. The remaining incoherence is concentrated in one place: the `apps/api` persistence
layer still wears the **clothes of a dual-backend system it no longer is** — six dispatch facades,
a whole `runtime_shape.py` decision module, and config types that resolve to a single value. That
is ~90% of a clean migration with a vestigial-abstraction tail that one focused sweep removes.

Answering the four questions directly:

1. **Intentional?** Split. `sibyl-core` yes (clean layering, intentional driver decomposition).
   `apps/api` persistence no — it's the strongest "half-migrated" signal in the tree. Retrieval is
   two engines wearing one design doc. The destination is clear and mostly reached; the gap is
   cleanup, not redesign.
2. **Boilerplate?** Yes, but concentrated, not pervasive — two big seams (~1,650 LOC of dead
   single-backend dispatch, plus a copy-pasted SurrealDB-coercion helper family) and a handful of
   dead modules. The large files are a mix of irreducible domain surface and a couple of real
   god-modules.
3. **Query management?** Safer than expected, less organized than ideal. 263 inline `execute_query`
   SurrealQL strings, no central builder — but **injection discipline is airtight** (every value
   bound via `$params`; interpolation is identifier/operator-only). Sprawl is real and cheap; the
   repository pattern is aspirational.
4. **Performance?** One headline and one systemic pattern. The **per-org query lock serializes
   every query for a tenant onto one websocket**, so the carefully-parallel retrieval fan-out
   re-serializes at the driver. Plus a pervasive load-the-whole-table-then-paginate-in-Python
   pattern, N+1 relationship writes, and no transactions anywhere (which is also the one
   **critical** correctness finding).

**Cohesion score: 7/10** (up from 6). **SurrealDB idiom score: 8/10** (up from 6) — native HNSW
KNN + BM25 FULLTEXT used correctly, parameter-bound throughout; held back by the connection lock and
the missing transaction support.

## What's genuinely solid (do not touch)

- **`sibyl-core` layering is intentional and one-way.** `services → backends/surreal`, zero
  back-references into `apps/*` in either direction. Real abstractions (`ActiveGraphRuntime`,
  `DedicatedSurrealClient`).
- **Driver decomposition is deliberate, not fragmentation.** The old monolith `driver.py` became
  `connection.py` (retry/classification) + a shared `DedicatedSurrealClient` + three thin domain
  subclasses (auth/content/graph). Clean and earns its keep.
- **No SQL injection anywhere.** All 263 `execute_query` sites bind user/runtime values via
  `$params`; the only interpolation is identifiers (static constants or `_IDENTIFIER_RE`-validated)
  and the genuinely-unbindable KNN `<|k,ef|>` operator. Candidate-fetch SurrealQL is DB-native
  (HNSW KNN + BM25 FULLTEXT), not Python-side row ranking.
- **Org isolation is sound and forge-proof.** Namespace-per-org (`org_<hex>`) applied via the SDK's
  `client.use()` RPC (not string interpolation), `group_id` always derived from a
  membership-validated JWT `org` claim, with redundant `WHERE group_id = $group_id` as
  defense-in-depth. The JWT org claim cannot be swapped to another tenant.
- **Observability went from absent to real.** Full `TelemetryRegistry` (counters/gauges/histograms
  with p50/95/99, Prometheus + JSON export) wired across request/query/job/LLM/search/queue paths,
  plus genuine request-correlation IDs in structlog.
- **There is a real E2E journey in CI** against a live SurrealDB with a seeded corpus; persistence
  clients are tested against embedded `memory://`, not mocked away.
- **Auth enforcement via `require_org_role` is uniform** at the router level — a genuine strength.
- **The D3 graph viz is high-craft** and performance-conscious, not a dumping ground. The CLI HTTP
  resilience layer (circuit breaker, pending-write buffering, idempotency, proactive refresh) is
  well-built and correctly **not** duplicated in web.

## Critical (fix before the next release)

### C1. Destructive multi-step cascades use no transactions — partial failure corrupts tenant data

`apps/api/src/sibyl/persistence/surreal/organization_runtime.py:984,1018,1057`,
`auth_runtime.py:703,3180`, archive restore in `content_archive.py` / `auth_archive.py`. **There is
no `BEGIN/COMMIT TRANSACTION` anywhere in the codebase.** `delete_org`, `delete_project_record` (4
sequential deletes), and archive restore run multi-statement cascades with no atomicity. A transient
failure mid-cascade leaves an org whose members/projects/api-keys are deleted but the org record
survives, or a deleted project leaving dangling `api_key_project_scopes`/`team_projects`/
`project_members`. For a multi-tenant memory product this is data-integrity-critical, and it's the
DR path, so a partially-restored namespace is especially dangerous. **Verified: confirmed
critical.**

**Fix:** wrap each same-namespace delete group in `BEGIN TRANSACTION; … COMMIT TRANSACTION;`. For
the cross-namespace `delete_org`, delete the `organizations` row first (immediately isolating the
org) then sweep child namespaces idempotently as background reconciliation, or write a deletion
tombstone a worker drains. At minimum make `delete_project_record`'s 4 statements one transaction.
Related: `delete_graph_data` (`graph_runtime.py:1160`) swallows the primary-path exception and
silently falls through to a non-transactional per-table DELETE loop — fix that fall-through too.

**Codex addition (verified):** transactions alone are not enough — `delete_org` doesn't even
**check the result status** of its child deletes (`organization_runtime.py:965,1029,1055,1059`), so
a child delete that errors is silently ignored regardless of wrapping. The fix needs both a
transaction wrapper **and** checked destructive-execute helpers (raise on a non-OK Surreal result),
which the shared coercion module from H5 should provide (`query_error` applied at every destructive
call site).

## High

Several reviewers independently flagged the same two issues (the facade and the lock) from different
lenses — that cross-surface consensus is signal, not noise. Deduped here.

### H1. The `apps/api` persistence layer is a vestigial single-backend dispatch facade (~1,650 LOC)

Flagged independently by the architecture, boilerplate, and persistence reviewers; hand-verified.
`apps/api/src/sibyl/persistence/{auth,organization,content,backups,settings,operations}_runtime.py`
exist purely to dispatch into exactly one backend. `auth_runtime.py:27` is
`_BACKEND_MODULES = {"surreal": "sibyl.persistence.surreal.auth_runtime"}` (single key) routing ~63
thin `async def x(**kw): return await _call_backend_export("x", **kw)` forwarders through
`_resolve_backend_export → import_module`. The same single-target dispatch is implemented **three
different ways** across the six files — a dict (auth), a one-element tuple
(`organization_runtime.py:233`), and a bare string constant (`content`/`backups`/`settings`/
`operations`) — which proves the dual-store abstraction was hollowed out file-by-file without
converging. `_unsupported_export`'s `NotImplementedError` machinery is unreachable. 28 caller files
pay an `import_module` + extra await frame and lose go-to-definition on every auth/org/content call.

**Impact:** the single strongest "accidental/transitional" signal in the codebase, directly
answering Q1 and Q2. **Fix (mechanical, high-leverage):** collapse the facades — re-export the
surreal symbols directly (`from sibyl.persistence.surreal.auth_runtime import *` with explicit
`__all__`) or point callers at the surreal modules and delete the shims, `_BACKEND_MODULES`,
`_resolve_backend_export`, `_unsupported_export`, and the `SIBYL_AUTH_STORE`-shaped error messages.
Pick one pattern if any seam is kept.

### H2. `runtime_shape.py` is a fully-elaborated dual-store decision layer over single-value types

`apps/api/src/sibyl/runtime_shape.py:7,13,22,26`. `RuntimeStore`/`AuthStore` are
`Literal["surreal"]`, so every helper is a constant: `default_auth_store()` returns `"surreal"`,
`uses_relational_auth()` returns `False`, `requires_surreal_support()` returns `True`. These launder
the `Literal` into a plain `str` param specifically so impossible values still typecheck, which
means `ty` cannot flag the dead arms. `config.py:146,150` carries `store`/`auth_store` as
single-value `Literal`s; `config.py:661,665` expose `fully_surreal_runtime`/`uses_relational_auth`
properties. The net effect: anyone reading config or CLI startup reasonably concludes a non-surreal
backend is still supported.

**Fix:** delete `runtime_shape.py` wholesale, inline the two truths (SurrealDB always required, auth
never relational), drop the `store`/`auth_store` fields (or freeze `store` to `Literal["surreal"]`
with a back-compat comment), and collapse the dead guards in `cli/migrate.py`,
`surreal_runtime_startup.py`, and `cli/up_cmd.py`. This is the keystone that turns the migration from
90% to actually clean.

### H3. The per-org query lock serializes every query for a tenant onto one websocket

Flagged by the performance, backend, and architecture reviewers; hand-verified at
`packages/python/sibyl-core/src/sibyl_core/backends/surreal/dedicated_client.py:118`. `_execute`
wraps the entire send-and-retry path — including the actual DB round-trip — in `self._query_lock`.
Each org funnels through one cached client, so **within an org the system can never run two
SurrealQL queries in parallel.** The retrieval pipeline's deliberate `asyncio.gather` fan-out over
4–6 candidate queries (`retrieval/search.py:500,887`) pays their latency sequentially; concurrent
MCP/REST requests for a hot org queue behind each other.

**This is a Principle-8 nerf baked into the driver.** The contended resource is the single websocket
per client, not SurrealDB (its wire protocol multiplexes responses by request id). **Fix by
scaling, not capping:** give each org a small pool of `DedicatedSurrealClient` connections and check
one out per query, or — if the current `surrealdb` Python SDK genuinely can't multiplex in-flight
queries on one socket — narrow the lock to the `connect()`/`_drop_client()` reconnect window only.
Do **not** "fix" this by reducing fan-out. Add a regression test that two concurrent reads on one
org client overlap.

### H4. Load-entire-table-then-paginate-in-Python across the content and backup layer

`apps/api/src/sibyl/persistence/surreal/content.py:566,963,1113`, `backups.py:309`. Every paginated
list reads the entire table/partition over the websocket, deserializes every row, sorts in Python,
and discards all but one page. For a crawl source with thousands of documents (the realistic RAG
case), page 1 transfers and parses the whole corpus — and because of H3, it also blocks every other
query for that org for the full read. Memory and latency scale with table size, not page size.

**Fix:** push `ORDER BY / START / LIMIT` into SurrealQL and run a separate `count()` for totals —
the codebase already does proper server-side counting in `graph_runtime.count_entities_by_type` and
`_graph_totals`; mirror it.

### H5. SurrealDB result-normalization/coercion helpers are copy-pasted across 5–8 files, and one already drifted

`auth_runtime.py:252,309`, `organization_runtime.py:70`, `content.py:122`, `auth.py`, `backups.py`,
`content_archive.py:181`, `auth_archive.py:194`. The `_normalize_record(s)`, `_query_error`,
`_coerce_uuid`, `_coerce_datetime`, `_utcnow` family is redefined in every persistence module.
**The drift is already a live correctness hazard:** the verifier confirmed a Surreal error shape
(bare nested result list) is detected in three modules but missed in `organization_runtime`, so an
org-scoped write could be treated as success when it errored. The `_utcnow` copies all strip tzinfo;
if one drifts you get cross-table timestamp skew.

**Fix:** extract one `sibyl_core.backends.surreal.records` module (next to `connection.py`) and
import it from both `apps/api` and `sibyl-core`; delete the per-file copies; route the
archive/backup code through it so it stops reaching past the facade into other modules' `_`-private
helpers.

### H6. Two divergent retrieval engines with different rankers

`retrieval/search.py:265,1871`, `hybrid.py:418`, `tools/context.py:701`, `tools/search.py:645`,
`api/routes/search.py:67`. The native `context_search` path (context packs) does RRF + arithmetic
boosts in-DB; the `hybrid_search` path (public `/api/search` and the MCP search tool) is a separate
engine running the 1,871-line `query_ranking` ranker (up to twice per query) plus a redundant
temporal layer. A query and a context-pack build for the same goal can return differently-ordered
results, and tuning one surface doesn't move the other. `retrieval-system.md` describes a pipeline
that matches neither path exactly.

**Fix:** pick one fusion+ranking core — route `hybrid_search`'s candidate fetch into the native
signal plan, or have `context_search` call the query ranker when a query is present. Collapse the two
in-Python RRF implementations (`search.py:1808`, `fusion.py:63`) into one, and update the doc to name
the real modules. (See M-tier for `bm25.py` — exported but production-unused — and `reranking.py` —
wired but missing its dependency — on this surface.)

### H7. Relationship writes are N+1 round-trips with no bulk path

`packages/python/sibyl-core/src/sibyl_core/services/graph.py:965,976`. Entities have a real batched
upsert; relationships only have `create_bulk()` that loops single `create_direct` calls — one query
+ one embedding call per edge — each a separate round-trip serialized behind H3's lock. Hit by
reflection promotion (`memory.py:324`), projection backfill (`projection/memory.py:395`), bulk
import (`api/routes/entities.py:1354`), and community storage.

**Fix:** add a `relates_to` bulk-upsert query and a `RelationshipManager.create_direct_bulk()`
mirroring the entity path; repoint the four call sites. Removes the N+1 without touching the lock.

### H8. Task/epic workflow is duplicated across REST and MCP and the two have drifted

`apps/api/src/sibyl/api/routes/tasks.py:408,421`, `epics.py:185`, vs
`packages/python/sibyl-core/src/sibyl_core/tools/manage.py:476,863`. The same state transitions are
re-implemented on both surfaces and **produce different system state**: REST adds entity locking,
idempotency, WebSocket broadcasts, and project-activity tracking that the MCP `manage()` path
silently skips. An agent completing a task via MCP races concurrent REST writers, emits no WS event
(web UI shows stale state), and epics started via MCP never update project activity timestamps.

**Fix:** extract the locked + broadcasting workflow wrapper into one shared service in `sibyl-core`
that both the REST handler and the `manage()` dispatcher call; delete the duplicate transition
bodies in `manage.py`. The MCP path should gain locking/broadcast by construction.

> **Superseded by a roadmap item.** The deeper fix is to remove the second entity entirely: an Epic
> is just a task with subtasks (`class Epic(Entity)` carries no scoping of its own, only a parallel
> `EpicStatus` and this duplicated workflow). Roadmap epic `epic_b913e99a537b` (northstar W14)
> collapses Epic into the task tree, so there is no parallel workflow left to keep in sync. H8's
> shared-service fix is the interim; the unification is the permanent one.

### H9. The 2,974-line web API client and 2,154-line hooks file have ~zero behavioral tests

`apps/web/src/lib/api.ts`, `hooks.ts`, `api.test.ts` (21 lines). The highest-risk client surface —
the 401→refresh→retry state machine and the normalizers that bridge backend records to frontend
types (`ApiKeyBackendRecord` → `ApiKey`) — is unverified. A backend field rename ships green.

**Fix:** a focused `api.test.ts` mocking `fetch` that pins (1) the refresh→retry→success and
→redirect paths, (2) each normalizer against a representative payload, (3) a couple of param
builders. Pin the state machine and normalizers, not 100% coverage.

### H10. API-key MCP auth still auto-promotes empty scopes to `["mcp"]` (Codex-surfaced miss)

The swarm verified that the *JWT* scopeless→MCP promotion was fixed and stopped there — but the
*API-key* path has the same hole. `apps/api/src/sibyl/auth/mcp_auth.py:42` auto-promotes an empty
stored scope list to `["mcp"]`. The create request defaults to `["mcp"]`, but a caller can pass an
empty list (`apps/api/src/sibyl/api/routes/auth.py:112`) which is stored as empty
(`auth_runtime.py:3628`) and then promoted at auth time. Net: an API key created with no scopes
silently gains MCP access. This is the same class of bug as the (fixed) JWT default-scope issue, on
a surface the migration left untouched.

**Fix:** require an explicit `mcp` scope for API keys too — reject empty scopes at creation, or
treat empty as "no access" at auth time rather than promoting to `["mcp"]`. Mirror the JWT fix in
`mcp_auth.py`.

## Medium (worth a dedicated cleanup pass)

**Architecture / boilerplate**

- `auth_runtime.py` (4,523 LOC) declares a repository base class used by exactly **one** subclass,
  then sprawls into ~135 free module functions / ~8 distinct repositories — a god-module that should
  split (`auth_runtime.py:676,722`).
- `services/graph.py` (2,735 LOC) mixes client lifecycle, generic entity CRUD, and Task/Epic/Project
  dashboard reporting (`get_project_summary` is a 113-line aggregator) — split the domain reporting
  out (`services/graph.py:123`).
- `schemas.py` (2,066 LOC) holds request/response models for ~13 unrelated domains in one file.
- 41 hand-rolled `try/except`-500 blocks duplicate the exact pattern the `handle_workflow_errors`
  decorator (6 uses) was built to eliminate (`api/decorators.py:31`).
- The repository abstraction exists but is half-adopted; ~18 `count()` shapes and N
  single-record-by-uuid lookups are hand-rolled across files instead of using shared clause
  builders.

**Dead code (retrieval) — spot-verified**

- `bm25.py` (380 LOC) is **exported but production-unused** (Codex correction — not "unimported"):
  package-exported at `retrieval/__init__.py:12` but called by no production path; real lexical
  search is SurrealDB FULLTEXT. Wire it in or remove it.
- `reranking.py` (414 LOC cross-encoder) is a **wired optional path missing its dependency**, not
  dead code (Codex correction): it IS imported by hybrid search (`hybrid.py:563`) but disabled by
  default (`config.py:231`), and `sentence-transformers` is absent from core deps
  (`pyproject.toml:30`), so `create_cross_encoder` raises `ImportError` if enabled. Meanwhile
  `use_enhanced=True` is advertised as default (`schemas.py:811`/`server.py:1133`) and `config.py:233`
  claims a "+33-40% accuracy" gain. Either add the dep and enable it, or stop advertising a path
  that cannot run.
- `fusion.py` is a third unused RRF implementation; `query_ranking.py` (1,871 LOC) is a
  benchmark-tuned scorer run up to twice per query; temporal ranking is in-memory-only and
  double-applied on the hybrid path. The LongMemEval replay uses its own duplicate ranker,
  undercutting the "same code as production" claim.

**Performance / correctness**

- HNSW embedding-dimension rebuild is fully scaffolded (`rebuild_index_concurrently`,
  `ConcurrentIndexDefinition`) but has **zero callers** (`schema.py:112`) — a dimension change after
  first bootstrap is still silently ignored at the schema level (the prior audit's dim-drift bug,
  now half-fixed: machinery exists, just unwired).
- Five module-global viz caches in `graph_communities.py:31` grow unbounded (TTL gates reads but
  never evicts, no write-side invalidation); `store_communities`/`get_entity_communities` do per-item
  N+1 writes/reads (`graph_communities.py:1904`).
- MCP path trusts stale `org_role` from the JWT claim instead of re-resolving membership live
  (`server.py:178`).
- MCP `add()` drops the epic linkage that REST `create_entity` passes (`server.py:843`) — divergent
  write paths.
- Core `CoreConfig` has no production guard for `memory://`/`surrealkv://` (only `apps/api` config
  does) (`config.py:142`).
- `manage()` remains a 21-action stringly-typed grab-bag with a self-admitted, unactioned
  deprecation notice (`tools/manage.py:332`).
- Two React Query mutations bypass the `fetchApi` wrapper with raw `fetch()`, losing auth-refresh
  (`hooks.ts:1667`).

**Migration tail**

- `requires_surreal_support` guards still wrap service selection despite always returning `True`
  (`cli/up_cmd.py:99`); `test_runtime_shape.py` asserts dual-store behavior with type-forbidden
  `"legacy"`/`"postgres"` literals, locking the dead code in place.
- Dead FalkorDB `MockGraphClient` / Postgres-Graphiti fixtures persist in core conftest and are
  duplicated into live test modules.

**Testing**

- Public `/health` is liveness-only — no readiness probe distinguishes "process up" from "can serve
  traffic" (`api/app.py:439`).
- Persistence E2E uses embedded `memory://`, which hides prod-only concurrency/index behavior.

## What the verifiers demoted (honesty about overstatements)

The adversarial pass demoted three high findings — recorded so we don't over-invest:

- **"`uses_relational_auth` gates 8 dead branches" → medium.** Real, but it's dead-code cleanup that
  rides along with deleting `runtime_shape.py` (H2), not its own high-priority item.
- **"Core graph unit tests were deleted, layer has no coverage" → low.** Overstated — the live-DB
  E2E job plus the `memory://` content/auth tests cover much of it. The real gap is the loss of a
  *fast* PR-gating unit loop for graph ops, which is narrower than "no coverage."
- **"Archive export/import non-atomic full-table" → medium.** Real and worth fixing (it's the DR
  path), but lower-frequency than the hot-path pagination in H4; folds into the same
  pagination + transaction work.

## Prioritized fix list (impact × urgency / effort)

1. **Wrap destructive cascades in transactions / compensation** (C1). `organization_runtime.py`,
   `auth_runtime.py`, archives, `graph_runtime.py:1160`. **Effort M.** Only correctness-critical
   item.
2. **Collapse the `*_runtime.py` facades and delete `runtime_shape.py`** (H1, H2). Removes ~1,650+
   LOC and the strongest half-migrated smell; mechanical, behavior-preserving. **Effort M.**

## Bottom line

This is no longer a fractured mid-migration codebase — it's a **mostly-finished migration with a
vestigial-abstraction tail**. The core library and the SurrealDB idiom are now genuinely good:
one-way layering, parameter-bound queries, native HNSW + FULLTEXT, real telemetry, real E2E, sound
and forge-proof multi-tenancy. The prior audit's scariest findings are largely closed — RBAC
fail-open is fixed, JWT→MCP scope promotion is fixed (the API-key path still needs H10), and
embedded-store in prod is default-denied (with an opt-in flag, plus a remaining core-config gap).

The work left is concentrated and mostly mechanical: delete the dual-backend dress (H1/H2), stop
serializing every tenant onto one socket (H3), centralize the copy-pasted SurrealDB coercion (H5),
push pagination into the DB (H4), and pick a single retrieval ranker (H6) — plus the one
correctness-critical gap, transactions on destructive cascades (C1). None of it is a redesign. The
destination is built; this is deburring the last 10%.
